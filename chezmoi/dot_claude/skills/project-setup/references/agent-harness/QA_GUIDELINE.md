# QA Guideline

このドキュメントは、{project-name} の機能変更に対する動作確認方針と実施手順をまとめる。lint、typecheck、format、build などのコード品質ゲートは責務外であり、ここではアプリケーションの振る舞いを確認する QA を扱う。

## 基本方針

- アプリ影響のある変更では、探索的テストを基本的に必ず実施する。
- Unit Test と E2E Test は変更内容に応じて選ぶ。ただし、検証対象の機能に対する Unit Test が不足している場合は、テストを追加して実行することで動作を検証する。
- E2E Test が既にある機能は、関連する E2E Test を実行する。検証に必要な E2E Test がなく、今後も回帰防止として残す価値がある場合は追加する。
- テストとして残すほど固定化すべきでない確認や、操作しながら違和感を探したい確認は、利用可能なブラウザ操作系ツールで実際にアクセス・操作して検証する。
- 認証があるアプリでは、QA / E2E 用に `DISABLE_AUTH=true` 等の認証 bypass を用意するか検討する。OAuth 等の外部認証は Agent が突破できないため、E2E の主要フローを認証 bypass で検証できるようにする。

## テスト戦略

変更内容に応じて、下記 3 種類のテストを選択し、実行する。アプリ影響のある変更では探索的テストを基本必須とし、Unit Test と E2E Test は検証対象・回帰防止の必要性に応じて追加または実行する。

### Unit Test

目的:

- 純粋関数、domain service、container hook、workflow の分岐を高速に検証する。
- 境界値、異常系、半正常系を明示的に固定する。

実施手順:

1. 変更対象に関連する既存テストを探す。
2. 正常系だけでなく、分岐・境界値・異常系が検証されているか確認する。
3. 不足している場合は Unit Test を追加する。
4. Unit Test を実行する。

実行コマンド:

```bash
pnpm test
```

### E2E Test

目的:

- 実際のブラウザで、ユーザーに見える振る舞いと主要フローを検証する。
- 重要な回帰防止シナリオを自動テストとして残す。

実施手順:

1. 変更内容を検証できる E2E Test がある場合は実行する。
2. ない場合、回帰防止として残す価値があるなら `e2e/` に Playwright test を追加する。
3. E2E Test を実行する。

実行コマンド:

```bash
pnpm test:e2e
```

認証があるアプリの E2E dev server は、必要に応じて `DISABLE_AUTH=true` で起動する。

### 探索的テスト

目的:

- 自動テストでは拾いにくい UI の違和感、遷移、非同期状態、エラー表示、実操作上の問題を見つける。
- アプリ影響がある変更に対して、実際にブラウザで触って振る舞いを確認する。

実施手順:

1. QA 用 dev server を起動する。
2. 利用可能なブラウザ操作系ツールで対象 URL にアクセスする。
3. 変更対象の画面・操作・正常系・異常系を実際に操作する。
4. 画面表示、URL、API 通信、console error、network error を確認する。
5. 発見した問題が回帰防止すべきものなら Unit Test または E2E Test に落とし込む。

## QA 実施チェックリスト

アプリ影響のある変更では、最低限次を確認する。

- [ ] 関連する Unit Test が十分か確認した。不足があれば追加して実行した。
- [ ] E2E Test が必要な変更なら、既存 E2E を実行するか、新規 E2E を追加して実行した。
- [ ] 探索的テストで、実際のブラウザ操作により変更箇所を確認した。
- [ ] console error と network error を確認した。
- [ ] 問題が見つかった場合、再現手順と期待結果を記録し、必要なら自動テストへ落とし込んだ。
